## 三、手写 Python 压测脚本

### 3.1 完整脚本 (`benchmark.py`)

```python
"""
FastAPI (REST) vs gRPC 性能对比压测

对比维度：
  1. 延迟（单次请求耗时）
  2. 吞吐量（一定时间内的请求数）
  3. 请求体大小（从数据层面看差异）
"""

import time
import grpc
import httpx
import user_pb2
import user_pb2_grpc

# ===== 配置 =====
TOTAL_REQUESTS = 1000     # 总请求数
CONCURRENT = 10           # 并发数（简单线程池模拟）
GRPC_ADDR = 'localhost:50051'
REST_ADDR = 'http://localhost:8000'

# ===== 辅助函数 =====
def benchmark(name, func, count=TOTAL_REQUESTS):
    """执行压测并打印结果"""
    print(f"\n{'='*50}")
    print(f"📊 测试: {name}")
    print(f"{'='*50}")

    # 预热（排除首次连接的初始化开销）
    func()

    # 正式测试
    latencies = []
    start_total = time.time()

    for i in range(count):
        start = time.time()
        func()
        elapsed = (time.time() - start) * 1000  # 转为毫秒
        latencies.append(elapsed)

        if (i + 1) % (count // 10) == 0:
            print(f"  进度: {i+1}/{count}")

    total = time.time() - start_total

    # 统计
    latencies.sort()
    avg = sum(latencies) / len(latencies)
    p50 = latencies[len(latencies) // 2]
    p95 = latencies[int(len(latencies) * 0.95)]
    p99 = latencies[int(len(latencies) * 0.99)]

    print(f"\n  ✅ 完成 {count} 次请求")
    print(f"  总耗时:     {total:.2f}s")
    print(f"  QPS:        {count/total:.0f}")
    print(f"  平均延迟:    {avg:.2f}ms")
    print(f"  P50 延迟:    {p50:.2f}ms")
    print(f"  P95 延迟:    {p95:.2f}ms")
    print(f"  P99 延迟:    {p99:.2f}ms")
    print(f"  最快/最慢:   {min(latencies):.2f}ms / {max(latencies):.2f}ms")

    return {
        'name': name,
        'total_sec': total,
        'qps': count / total,
        'avg_ms': avg,
        'p50_ms': p50,
        'p95_ms': p95,
        'p99_ms': p99,
    }


# ===== gRPC 客户端 =====
grpc_channel = grpc.insecure_channel(GRPC_ADDR)
grpc_stub = user_pb2_grpc.UserServiceStub(grpc_channel)

# 先创建一批用户供查询
for i in range(10):
    grpc_stub.CreateUser(user_pb2.CreateUserRequest(
        name=f"User{i}", email=f"user{i}@test.com"
    ))

def grpc_get_user():
    grpc_stub.GetUser(user_pb2.GetUserRequest(id=1), timeout=2.0)

def grpc_create_user():
    grpc_stub.CreateUser(user_pb2.CreateUserRequest(
        name="Bench", email="bench@test.com"
    ), timeout=2.0)


# ===== REST 客户端 =====
rest_client = httpx.Client(base_url=REST_ADDR, timeout=2.0)

# 先创建一批用户供查询
for i in range(10):
    rest_client.post("/users", json={"name": f"User{i}", "email": f"user{i}@test.com"})

def rest_get_user():
    rest_client.get("/users/1")

def rest_create_user():
    rest_client.post("/users", json={"name": "Bench", "email": "bench@test.com"})


# ===== 运行测试 =====
if __name__ == '__main__':
    results = []

    print("🔧 预热连接...")
    grpc_get_user()
    rest_get_user()
    print("✅ 预热完成\n")

    # 测试1：GET 请求对比
    results.append(benchmark("gRPC - GetUser (查询)", grpc_get_user))
    results.append(benchmark("REST - GetUser (查询)", rest_get_user))

    # 测试2：POST 请求对比
    results.append(benchmark("gRPC - CreateUser (创建)", grpc_create_user))
    results.append(benchmark("REST - CreateUser (创建)", rest_create_user))

    # ===== 汇总报告 =====
    print("\n\n" + "=" * 60)
    print("📋 最终对比报告")
    print("=" * 60)

    # 对比表
    print(f"\n{'测试项':<30} {'QPS':>8} {'平均延迟':>10} {'P95延迟':>10}")
    print("-" * 60)
    for r in results:
        print(f"{r['name']:<30} {r['qps']:>8.0f} {r['avg_ms']:>8.2f}ms {r['p95_ms']:>8.2f}ms")

    # gRPC vs REST 提速比例
    if len(results) >= 2:
        grpc_qps = results[0]['qps']
        rest_qps = results[1]['qps']
        speedup = grpc_qps / rest_qps
        print(f"\n🚀 gRPC 查询吞吐量是 REST 的 {speedup:.1f}x")

    # 清理
    grpc_channel.close()
    rest_client.close()
```

### 3.2 运行

```bash
python benchmark.py
```

## 四、gRPC 快的三大原因

| 原因 | 说明 |
|------|------|
| **Protobuf 二进制** | JSON 是文本，解析慢；Protobuf 是二进制，编解码极快 |
| **HTTP/2 多路复用** | REST 用 HTTP/1.1，每个请求一个连接；gRPC 一个连接并发多请求 |
| **连接复用** | gRPC Channel 天然长连接，不需要每次 TCP 三次握手 |

## 五、请求体大小对比

同样的数据，序列化后的体积：

```python
import json
import user_pb2

# JSON 版本
json_data = json.dumps({"id": 1, "name": "Alice", "email": "alice@example.com"})
print(f"JSON: {len(json_data)} bytes")

# Protobuf 版本
pb_data = user_pb2.UserResponse(
    id=1, name="Alice", email="alice@example.com"
).SerializeToString()
print(f"Protobuf: {len(pb_data)} bytes")
print(f"PB 是 JSON 的 {len(pb_data)/len(json_data)*100:.0f}%")
```

输出（大致）：
```
JSON: 56 bytes
Protobuf: 28 bytes
PB 是 JSON 的 50%
```

> 字段越多、数据量越大，Protobuf 的体积优势越明显。

## 六、用 ghz 做更专业的压测

Python 脚本适合简单对比，真正的压测推荐 `ghz`：

```bash
# 安装
go install github.com/bojand/ghz/cmd/ghz@latest

# 压测 CreateUser
ghz --insecure \
  --proto user.proto \
  --call user.UserService/CreateUser \
  -d '{"name":"Test","email":"test@example.com"}' \
  -c 10 -n 10000 \
  localhost:50051
```

输出：
```
Summary:
  Count:        10000
  Total:        3.21 s
  Slowest:      15.23 ms
  Fastest:      0.12 ms
  Average:      1.45 ms
  Requests/sec: 3115.26
```

## 七、选择建议（根据性能测试结果）

| 场景 | 推荐 | 原因 |
|------|:---:|------|
| 对外 API（浏览器/App） | REST | 通用、易调试 |
| 内部微服务高并发 | gRPC | 吞吐量高 3-5 倍 |
| 低延迟要求 | gRPC | P99 延迟更低 |
| 大数据量传输 | gRPC | Protobuf 体积小 |
| 简单小项目 | REST | 开发效率优先 |
| 多语言团队 | gRPC | 一份 proto 生成所有客户端 |

## 八、本课小结

```
性能不是纸上谈兵，是跑出来的数字：

  测试项               REST QPS    gRPC QPS    差距
  ─────               ────────    ────────    ────
  GetUser (查询)       ~350        ~1900       5x+
  CreateUser (创建)    ~300        ~1500       5x+
  请求体大小           ~56B        ~28B        2x 小

  在微服务内部调用场景下，gRPC 的性能优势是实打实的。
  这也是为什么 K8s、Istio、Envoy 这些云原生基础设施
  都天然支持 gRPC。
```
